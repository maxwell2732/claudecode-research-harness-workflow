New Harness V2 / Phase 73

何がどう変わるのか

あなたが細かく計画や検証を抱える運用から、AI 側が仕様正本・実装計画・検証ゲートを先に組み、あなたは重要判断だけを見る運用へ変わる。

差分を見る 対応ツールを見る

一番大きな変化

以前は「導入できたか」「コードが通ったか」に寄りがちだった。V2 / Phase 73 では「その tool で agent が正しい流れを始め、検証できるか」までを合格条件にする。

今まで

  • Claude Code plugin が中心で、Codex や OpenCode は別導線。
  • あなたが計画の妥当性と検証結果を多めに見る。
  • 「対応している」と「試せそう」が混ざりやすい。
  • PR ready と release ready が心理的に近くなる。

これから

  • 最初に使う tool を選び、その tool 専用の導入に入る。
  • AI が spec、Plans、TDD、review、PR closeout を先に組む。
  • support tier で、対応済み・候補・未対応を分ける。
  • PR と release を別ゲートにし、公開物だけ追加検証する。

新しい作業フロー

Phase 73 は「共通の作業スキルを各AIツールへ薄く配る設計」を、Harness の stage gate に変換する。ポイントは「速くするが、証拠の段を抜かない」こと。

検証・調査 参考元の設計と各 tool の現在地を確認し、unknown を unknown のまま残す。
計画確定 spec.md と Plans.md に、scope、lane、DoD、support tier を固定する。
実装 TDD 必要な箇所は先に failing test を作り、実装は gate に沿って進める。
サブエージェント 調査、実装、仕様レビュー、品質レビューを分けて並走する。
結果検証 bootstrap、skill trigger、migration、adapter smoke を確認する。
PR / Release PR evidence と release evidence を分け、公開 claim は検証済みに限定する。

あなたの使い方はどう変わるか

依頼は短くてよくなる

「これ作って」「この issue 進めて」で、AI 側が spec.md と Plans.md を見て lane、DoD、TDD要否、レビューの深さを組む。あなたは毎回、細かい実装手順を作らなくてよくなる。

見るべきものが減る

あなたが見るのは、support tier、未観測データ、テスト結果、review verdict、PR / release の残リスク。コード全部を読むより、正本と証拠が矛盾していないかを見る運用に寄る。

導入は tool 起点になる

新規ユーザーには「Claude ならこれ」「Codex ならこれ」「OpenCode ならこれ」と出す。既存ユーザーには backup、migration impact、rollback を先に見せる。未検証 tool は候補として扱う。

具体的なユースケース

新機能を作る

AI がまず仕様正本と Plans の差分を作る。必要なら TDD red evidence を作り、サブエージェントが実装とレビューに分かれる。

あなたは scope と残リスクを見る。

Codex で使いたい

Claude から反映するだけでなく、Codex direct plugin path を検証対象にする。Codex app は CLI と分けて smoke する。

導入の回り道が減る。

既存ユーザーを移行する

いきなり削除しない。現在の plugin/cache/skills/harness-mem state を inventory し、backup と rollback を出す。

互換性の不安が減る。

一言でいうと

New Harness は、あなたの「考える時間」を奪うのではなく、判断に必要な材料を先に並べる環境になる。速く攻めるために、対応済みと未検証を分ける。
